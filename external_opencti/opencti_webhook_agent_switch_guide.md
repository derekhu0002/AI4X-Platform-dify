# OpenCTI Webhook Switch Guide (Agent Boundary)

## Scope
Switch OpenCTI webhook target from ai4sec_opencti_mcp legacy ingress to Dify native Agent/Workflow webhook ingress.

Recommended asset: import and publish `DifyAgentWorkflow/ai4sec_opencti_webhook_workflow.yaml` as the dedicated OpenCTI webhook workflow, separate from the user-facing unified workflow.

## Target Endpoint
- Method: POST
- URL: Use the webhook URL generated by Dify application/workflow publish page (for example `http://<dify-host>/e/<hook_id>`)
- Auth: not required by current architecture decision
- Payload: STIX 2.1 bundle (type=bundle, objects non-empty)

## Migration Steps
1. Export current OpenCTI webhook configuration for rollback reference.
2. In Dify, import `DifyAgentWorkflow/ai4sec_opencti_webhook_workflow.yaml` and publish that dedicated workflow.
3. Update OpenCTI webhook URL to the Dify-generated URL above.
3. Keep existing event filters unchanged for the first cutover.
4. Send one test bundle and verify HTTP 200 with accepted=true.
5. Confirm Dify execution history shows the workflow run was triggered by webhook.
6. Disable legacy target if it still points to ai4sec_opencti_mcp.

## Validation Checklist
- Dify dedicated webhook workflow receives bundle and triggers workflow run.
- Legacy ai4sec_opencti_mcp webhook route is no longer exposed.
- Downstream workflow inputs include `entry_mode/selected_flow/bundle` mapping used by runtime nodes.
