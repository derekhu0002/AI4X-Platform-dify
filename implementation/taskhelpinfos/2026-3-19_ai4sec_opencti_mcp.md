## 1. LLM执行摘要

- 当前任务要完成的是移除 `ai4sec_opencti_mcp` 及相关代码，原因是投影能力已下沉到各 Agent 流中，回写能力也不再需要。
- 首要修改对象是 `mcp/opencti_mcp/app/` 下的 FastAPI 实现，以及仍显式依赖 `OPENCTI_MCP_URL` 的统一工作流入口。
- 不允许越界把当前任务扩展成新的 OpenCTI SDK 设计或新的通用查询层改造；已知方向是删除兼容服务，而不是重建另一层中间件。
- 最关键验收条件是仓库运行时不再依赖该 MCP 服务，且历史兼容接口不再作为对外依赖面存在。
- 由于 `ai4sec_agent` 在架构中仍通过关系 1082 调用 `ai4sec_opencti_mcp`，删除后必须显式记录统一工作流与架构描述的失配风险。
- `TechnologyLayer` 目前仍把该服务视为 Dify 到 OpenCTI 的统一接入面，因此运行时下线后文档会滞后，这属于可预期后续清理项。
- `.env` 与 `.env.example` 目前保留多项 `OPENCTI_MCP_*` 配置，是需要同步收缩的运行时残留。
- 仓库中的 webhook 工作流与 VS1 独立威胁建模工作流已有直连 OpenCTI 的事实，可作为“该服务可移除”的已确认上下文。
- 当前主要风险是误删共享的 OpenCTI 查询逻辑或配置样例，导致仍存活的工作流无法访问 OpenCTI。

## 2. 已确认事实

- 架构元素 `ai4sec_opencti_mcp`（1215）对应代码路径 `mcp/opencti_mcp/app/main.py`、`service.py`、`settings.py`、`models.py`。
- `ApplicationLayer`（1209）中关系 1082 定义 `ai4sec_agent`（1214）访问 `ai4sec_opencti_mcp`（1215），关系 1083 定义 `ai4sec_opencti_mcp` 访问 OpenCTI 平台。
- 元素 1215 的描述已明确：VS1 独立威胁建模工作流（1436）和 WEBHOOK_AGENT（1277）已直接访问 `OPENCTI_GRAPHQL_URL`，运行时不再依赖该服务；仅 `ai4sec_agent`（1214）继续通过 `OPENCTI_MCP_URL` 调用它。
- 原则 `全局情报底座：OPENCTI 平台`（1228）要求内部情报由 OpenCTI 统一承载，删除 MCP 不应改变 OpenCTI 作为事实底座的角色。
- 原则 `业务流程中所有数据都必须是OPENCTI平台中的STIX2.1标准数据`（1226）要求即便移除中间服务，也不能破坏 STIX 2.1 语义一致性。
- `README.md`、`.env`、`.env.example` 当前仍把 OpenCTI MCP 作为正式启动与配置单元暴露。

- 必须遵守 `Progressive Disclosure`：先处理 MCP 服务与运行时入口，再把文档、测试、架构模型同步留给后续批次。
