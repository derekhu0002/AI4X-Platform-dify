## 1. LLM执行摘要

- 本任务目标是把当前仅路由的 Dify 工作流扩展为“受理-查询-研判-写回-通知-反馈”闭环。
- 首要修改对象是工作流编排与 HTTP 节点调用链，优先打通 VS4 主场景。
- 不允许越界到其余 value stream 分支复制，本次只完成主场景闭环。
- 反馈环节必须触发人工复核，人工确认后才允许写回 OpenCTI Note/Opinion/Relationship。
- 写回和通知优先使用当前 Dify 版本可用的 HTTP 节点能力。
- 关键验收条件是闭环端到端可执行且可回查，不是仅完成路由演示。
- 人工节点权限和 SLA 尚未确认，需作为显式前置风险管理。
- KPI 口径与监控阈值未确定，暂不作为本任务完成阻断条件，但需标注待确认。

## 9. 风险、阻塞与缓解措施

- 技术风险：工作流节点增加导致时延上升。缓解：分阶段压测并优化节点超时。
- 依赖风险：人工节点权限不足。缓解：预设降级路径并提前完成权限验证。
- 语义风险：写回字段无白名单导致污染。缓解：先启用最小白名单，再逐步扩展。
- 范围风险：团队将“主场景打通”误解为“全分支完成”。缓解：在验收标准中显式声明范围边界。
