## 1. LLM执行摘要

- 当前任务的目标是把 report 查询从“先拉全量再匹配”改成“优先使用手头 report id 或精确 name 的单对象查询”。
- 首要修改对象是 `ai4sec_opencti_mcp` 对 OpenCTI report 的查询与匹配逻辑；如果该能力已迁移到工作流节点，则同样适用于该节点实现。
- 任务说明已经给出严格约束：有 ID 优先按 `id` 查；没有 ID 时才按精确 `name` 查；live mode 下既不是标准 report ID 也不是精确 name 时，不允许 fuzzy fallback。
- 兼容层仍需保留本地测试 bundle report ID 到 live 标识的 alias 映射能力，但该兼容层不能退化为“查全量再模糊比对”。
- 单对象查询限额与分页边界必须继续遵守应用层合同，不得额外扩大查询范围。
- 所有 report 查询结果仍需保持 STIX 2.1 语义，并继续以内部 OpenCTI 作为唯一事实来源。
- 本任务不要求扩展到 Notification、统一工作流、容器编排或无关对象类型查询。
- 最关键验收条件是：给定 report ID 或精确名称时，系统只做单对象/定向查询并返回目标 report 及其必要上下文，而不是全量回表。
- 当前主要风险是部署中的 OpenCTI 版本实际支持的 report 精确查询字段、内部 ID 类型和 alias 映射来源仍需结合代码与运行环境核实。

## 4. 约束与边界

- 必须遵守 `Progressive Disclosure`：默认只查询完成当前 report 研判所需的最小对象与字段，不得为了兼容而回退到全量列表扫描。
- 必须遵守 `Separation of Concerns`：输入识别、OpenCTI 精确查询、alias 兼容、字段投影、结果回传要分别处理，不得把模糊兜底散落到多个层次。
- 必须遵守任务给出的强制规则：有 ID 优先按 `id`；无 ID 才按精确 `name`；live mode 非法输入不允许 fuzzy fallback；alias 仅作为兼容层。
- 必须保持不变的边界：OpenCTI 仍是唯一事实底座，查询结果仍符合 STIX 2.1，现有应用层查询配额与分页边界不被放宽。
- 明确禁止：先拉全量 report 再做本地匹配、在 live mode 对任意自由文本做模糊兜底、把 alias 兼容层实现成隐藏的全量扫描。
- 明确禁止：顺手扩展到无关对象类型、修改 Notification 合同、扩大 OpenCTI 查询深度或分页上限。

| OpenCTI 实际 GraphQL 查询入口与任务说明不完全一致 | 可能导致按 `id` / `name` 精确查询落地失败 | 先以当前部署版本做最小验证，再锁定最终查询参数 |
