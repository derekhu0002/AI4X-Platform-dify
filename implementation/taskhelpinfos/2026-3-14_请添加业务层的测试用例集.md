## 1. LLM执行摘要

- 当前任务要产出一套面向真实系统人工验收的业务层测试用例，不是自动化单元测试或集成测试。
- 首要修改对象是 `tests\validation` 目录下的新建 Markdown 文档与汇总表。
- 测试设计必须按 capability 视角组织，而不是按业务角色拆分测试套件。
- 测试数据必须基于 4 条价值流用户故事场景构造，并显式使用 STIX 2.1 对象语义描述关键输入数据。
- 测试入口与交互边界必须保持在 Dify、OpenCTI、Notification MCP 的既有职责分工内，不能把业务层测试写成底层部署或接口实现验证。
- 最关键验收条件是：每条 capability 都有清晰的通过/不通过判定，并且能暴露业务层设计与现网能力的差距。
- 测试用例需要包含人工验收记录模板，但不要求量化评分、截图留证或审批签字。
- 现网访问前提已给出：Dify 使用 `http://localhost/apps`，OpenCTI 使用 `http://localhost:8080/dashboard`，OpenCTI 数据通过平台导入功能导入。
- 任务边界内需要把人工验收用例与已有自动化测试目录分开管理；禁止把该任务转写为 Python 自动化测试实现。
- 主要风险是业务层 capability 与现网当前能力可能不完全一致，涉及“当前暴露差距”而不是“顺手补实现”。

## 4. 约束与边界

- 必须遵守原则：Progressive Disclosure；Separation of Concerns；业务流程中所有数据都必须是OPENCTI平台中的STIX2.1标准数据（1226）；全局情报底座：OPENCTI 平台（1228）；统一智能入口：DIFY Agent (ai4sec_agent)（1229）；通知预警标准化通道（1230）。
- 必须保持不变的模块或边界：DIFY（1212）、ai4sec_agent（1214）、ai4sec_opencti_mcp（1215）、OpenCTI platform（1216）及 Notification 通知边界的既有职责不能因测试文档而被改写。
- 明确禁止的实现方式：不要把任务实现成 `tests\unit`、`tests\integration`、`tests\e2e` 下的自动化脚本；不要补写底层部署、SMTP、GraphQL 细节；不要为弥补现网差距而追加产品功能开发。
- Progressive Disclosure 强制要求：测试文档需要按“测试范围/前置条件 -> 测试数据 -> capability 级用例 -> 人工记录模板 -> 差距暴露汇总”分层展开，避免把所有细节堆在单一表格中。
- Separation of Concerns 强制要求：业务能力验收、环境准备、测试数据、人工记录模板、现网差距说明必须分节表达，不能把业务验收标准与底层技术实现混写。
- 与架构不一致时的处理边界：当前任务只负责显式暴露差距，不负责在测试文档之外新增补偿逻辑或修改业务架构定义。
