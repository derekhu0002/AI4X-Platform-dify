## 4. 约束与边界

- 必须遵守原则：`业务流程中所有数据都必须是OPENCTI平台中的STIX2.1标准数据`（1226）、`全局情报底座：OPENCTI 平台`（1228）、`统一智能入口：DIFY Agent (ai4sec_agent)`（1229）、`通知预警标准化通道`（1230）。
- 必须遵守基线原则：`Progressive Disclosure`。文档结构应先从价值流目标、业务角色、业务能力开始，再逐步展开业务服务、流程、对象和协作边界，禁止一开始直接落到接口、脚本、部署或数据表。
- 必须遵守基线原则：`Separation of Concerns`。业务层仅描述业务责任、业务能力与流程，不把应用层组件职责、MCP 接口语义、OpenCTI 部署参数或通知渠道配置混写为业务元素。
- 统一能力地图标准必须采用三层结构：一级为共性能力域，建议统一为“情报接入与标准化、情报研判与风险评估、响应协同与处置编排、知识沉淀与持续优化、治理与绩效管理”；二级为跨价值流复用的通用业务能力；三级为面向具体价值流场景的业务服务或业务活动。
- 每个二级能力至少要绑定：服务对象、触发条件、关键输入输出、责任角色、对应价值流、至少 1 个可度量 KPI；禁止只列能力名而无边界与产出定义。
- 业务对象必须按“先粗后细”落地：先定义一级对象簇，例如情报任务、情报资产/上下文、风险结论、响应动作、证据链、通知与反馈，再视场景细化对象属性、状态和关系。
- 角色建模必须补充 RACI；关键端到端业务流程必须提供泳道图；关键能力域必须提供 KPI/度量项，三者均属于本次文档必备内容。
- 必须保持不变的边界：`design/tasks/taskandissues_for_LLM.md` 中的原始任务行；`design/KG/SystemArchitecture.json` 中已有应用层与战略层事实；`config/opencti_webhook_signal_template.ejs` 与 `external_opencti` 下运行配置。
- 明确禁止的实现方式：直接把 `ai4sec_agent`、`ai4sec_opencti_mcp`、`OpenCTI platform`、`Notification MCP` 当作业务能力名称；直接输出实现代码、接口定义、容器编排或环境变量说明；绕开 OPENCTI 设计外部平台直连 Agent 的业务流程。
- 若需要表达新的业务角色、业务能力、业务服务、业务流程、业务对象，而 KG 中不存在对应元素，应明确标注 `需新增架构元素`，不得伪造已存在 ID。

## 9. 风险、阻塞与缓解措施

| 风险/阻塞 | 类型 | 影响 | 缓解措施 |
| --- | --- | --- | --- |
| 业务层现状为空，缺少可复用业务元素 | 信息缺口 | 容易导致设计粒度不稳定 | 先从 4 条价值流抽取共性，再分层定义角色、能力、流程、对象 |
| 角色细分要求明确，但 KG 中缺少独立角色元素 ID | 架构缺口 | 后续落模时可能出现命名冲突或职责重叠 | 在文档中先定义角色及职责，并显式标注 `需新增架构元素` |
| 文档模板与命名规范未给出 | 交付风险 | 输出样式可能与团队预期不一致 | 在文档开头写明采用的命名与结构假设，提交时提醒评审确认 |
| 统一能力地图若划分过细或过散 | 设计风险 | 会削弱跨价值流复用，导致能力与服务边界混乱 | 固定采用“三层能力地图”标准，先共性后场景，避免直接从活动清单反推能力 |
| 业务对象若一开始细化过深 | 设计风险 | 容易在缺少业务共识时产生大量低复用对象 | 严格采用“先粗后细”，先沉淀一级对象簇，再对关键场景对象做最小必要细化 |
| RACI、泳道图、KPI 缺失或彼此不一致 | 治理风险 | 会导致设计可执行性和可考核性不足 | 以同一角色清单和能力地图为基准统一生成三类治理视图，并交叉校验责任与指标归属 |
| 容易把应用层平台能力误写成业务能力 | 设计风险 | 破坏分层边界，降低业务模型可复用性 | 在“约束与边界”中明确禁止项，先写业务意图再写协作边界 |
| 仅有任务描述，缺少显式验收示例 | 验收风险 | 审阅标准可能不一致 | 用可核对清单定义验收条件，确保价值流覆盖与业务元素完整性可检查 |

## 10. 下一步建议

- 先在 `design/businesslayer` 内确定文档文件名与目录结构，并把这一命名假设写入文档开头。
- 再按“角色 -> 统一能力地图 -> 业务流程 -> 业务对象 -> RACI/泳道图/KPI -> 跨层边界”的顺序完成业务层设计。
- 若后续需要同步更新 KG，请单独发起建模任务，不要与当前文档交付混做。
