## 4. 约束与边界

- 必须遵守 `Progressive Disclosure`：执行顺序固定为“输入报告标识 -> 匹配报告 -> 查询 OpenCTI -> 过滤字段 -> LLM 威胁分析 -> 展示结果与下载”，禁止一次性暴露底层全量原始对象。
- 必须遵守 `Separation of Concerns`：Dify Workflow 负责用户交互和编排，ai4sec_opencti_mcp 负责 STIX 查询与过滤，OpenCTI Runtime Stack 负责事实存储与检索，用户故事文件负责设计追踪；禁止职责混写。
- 必须遵守 Principle：业务流程中所有数据都必须是 OPENCTI 平台中的 STIX2.1 标准数据（1226）。
- 必须遵守 Principle：全局情报底座：OPENCTI 平台（1228）；所有分析输入都必须从内部 OpenCTI 查询得到。
- 必须遵守 Principle：统一智能入口：DIFY Agent (ai4sec_agent)（1229）；不得绕过 Agent 直接把查询链路暴露给用户。
- 必须保持不变的模块边界：现有统一工作流资产、非 VS1 用户故事资产、OpenCTI 运行栈角色边界、MCP 的查询过滤职责边界。
- 明确禁止的实现方式：在现有统一工作流基础上拆分代替新增独立 YAML；更新现有 VS1-E2E_威胁建模_端到端用户故事代替新增衍生故事；把 LLM 分析结果直接写回 OpenCTI；在独立 Workflow 中接入 Notification MCP 完整通知分支；跳过精确匹配直接做模糊匹配。

## 9. 风险、阻塞与缓解措施

- 风险：独立 Workflow YAML 的命名和入口约定未在 KG 中明确。
  缓解措施：沿用现有工作流资产命名风格，无法确认时仅在文件级保持 VS1 语义清晰。

- 风险：双模糊匹配的具体算法和排序规则缺失，可能导致查询结果不稳定。
  缓解措施：先固定“精确匹配优先”的主规则，模糊匹配部分明确标记为需人工确认的实现细则。

- 风险：如果复用现有统一工作流过深，容易违反“新增独立 YAML”边界。
  缓解措施：把复用控制在共享工具或 MCP 接口层，避免直接在统一工作流中拆分主链路。

- 风险：若 MCP 不做字段过滤，会把无关 STIX 字段和关系噪声带入 LLM 分析。
  缓解措施：严格按对象 1282 的要求在 MCP 层裁剪字段，并把过滤逻辑与分析逻辑分离。

- 风险：下载交互方式未在 KG 中明确，可能影响最终会话体验落地。
  缓解措施：保持输出为标准 JSON 文件，并把具体下载入口作为实现细节留在 Workflow/UI 层处理。
