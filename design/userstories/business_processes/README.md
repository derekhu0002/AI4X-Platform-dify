## 业务流程设计方法论 (Business Process Design Methodology)

我们在设计业务流程时，遵循以下标准方法论，以确保流程的清晰性、完整性和可执行性。

### 1. 设计原则 (Design Principles)
*   **原子性 (Atomicity)**: 每个流程应聚焦于解决单一核心业务问题，避免过于庞大。
*   **端到端 (End-to-End)**: 流程必须有明确的起点（触发条件）和终点（交付成果）。
*   **角色明确 (Role Clarity)**: 必须明确每个步骤的执行者（如：用户、API网关、核心引擎、插件）。
*   **数据驱动 (Data Driven)**: 明确步骤之间流转的数据对象（输入/输出）。所用到的数据对象必须引用自 `design/businessObjects`。

### 2. 流程结构标准 (Structure Standard)
每个业务流程文档应包含以下核心要素：

1.  **流程元数据 (Metadata)**
    *   **ID**: 唯一标识符。
    *   **Owner**: 流程负责人。
    *   **触发条件 (Trigger)**: 什么事件启动了该流程？
    *   **前置条件 (Pre-conditions)**: 执行前必须满足的状态。

2.  **角色/泳道 (Swimlanes)**
    *   **User/Operator**: 人类交互点。
    *   **Interface Layer**: API或CLI入口。
    *   **Orchestrator (Core)**: 决策与调度中心。
    *   **Execution Layer (Plugins/Workers)**: 实际干活的组件。
    *   **Data Layer**: 数据读写操作。

3.  **流程图 (Diagram)**
    *   使用 Mermaid `sequenceDiagram` 或 `flowchart` 绘制。
    *   展示组件交互时序。

4.  **详细步骤说明 (Step-by-Step Description)**
    *   对关键步骤的详细逻辑描述，包括异常处理路径。

### 3. 生命周期管理 (Lifecycle Management)
*   **Draft**: 草案阶段，仅定义核心路径。
*   **Review**: 架构师评审，确认组件交互可行性。
*   **Implemented**: 代码已实现，测试用例已覆盖。
*   **Deprecated**: 流程已废弃或被新流程取代。
